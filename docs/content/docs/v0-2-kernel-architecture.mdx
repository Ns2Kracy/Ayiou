# 目标

v0.2 的目标不是“做多 Bot 编排平台”，而是把 Ayiou 收敛成一个足够清晰的单 Bot 内核：

- 一个应用里只有一个由开发者定义的 `Bot`
- 不同消息渠道通过 adapter 接入同一个 Bot runtime
- 回复时根据事件来源路由回原渠道，即“从哪来到哪去”
- 插件只面对统一模型，不关心事件来自 Console、OneBot 还是其他平台

这比多 Bot 管理更符合 Ayiou 当前的复杂度边界。多 Bot 会引入 catalog、definition、runtime graph、跨 Bot 配置与生命周期协调，但这些能力现在没有明确收益，反而会让插件模型变重。

v0.2 追求的结果是：

- 开发者写插件时，看到的是一套 `RuntimePlugin` 模型
- 运行时管理插件时，也使用同一套模型
- 多渠道只是 adapter/source 差异，不是 Bot 架构分叉

## 借鉴什么，不借鉴什么

### 借鉴 NoneBot2

- `Matcher` 风格的事件过滤与分层匹配
- `Rule / Permission / Priority / Block` 的执行管线思想
- Adapter 与平台能力解耦的方向
- 依赖注入让插件接口保持简洁

### 不直接照搬 NoneBot2

- 不把核心抽象做得过度动态，避免大量隐式依赖和调试困难
- 不让会话控制强绑定某一种 matcher 语义
- 不让插件编排过度依赖“全局注册 + 隐式状态”

### 借鉴 AstrBot

- 插件配置和管理体验更接近产品化
- 装饰器式 filter 对消息型插件很顺手
- Session Controller 这类高层能力有实际开发价值
- 更强调插件实例管理，而不是只有“插件类型”

### 不直接照搬 AstrBot

- 不把运行时设计过度绑定到上层产品形态
- 不把过多行为藏进装饰器和魔法属性里
- 不把平台特性直接渗透到通用插件主路径

## v0.2 设计原则

1. 单一插件模型
2. 单 Bot，多渠道 adapter
3. 通用优先，平台扩展显式进入
4. 先有执行管线，再做语法糖
5. 配置、权限和能力协商先进入运行时；存储、会话、定时任务和观测性等服务边界后移

# 核心设计

## 1. 统一为 RuntimePlugin

旧设计里曾同时存在：

- 面向消息分发的 `Plugin<C>`
- 面向生命周期管理的 `ManagedPlugin`

v0.2 不再保留这两套平行体系，而是统一为：

- `RuntimePlugin`：插件行为、生命周期、配置应用、事件处理、健康状态
- `RuntimePluginEngine`：单 Bot 内的插件运行时与 handler 调度器

一个简化轮廓如下：

```rust
#[async_trait::async_trait]
pub trait RuntimePlugin<C: Sync + 'static>: Send + Sync + 'static {
    fn kind(&self) -> &str;
    fn meta(&self) -> PluginMetadata;
    fn manifest(&self) -> RuntimePluginManifest;
    fn declared_handlers(&self) -> Vec<HandlerDecl>;

    async fn init(&mut self, services: RuntimePluginServices<C>) -> anyhow::Result<()> { Ok(()) }
    async fn start(&mut self, services: RuntimePluginServices<C>) -> anyhow::Result<()> { Ok(()) }
    async fn stop(&mut self) -> anyhow::Result<()> { Ok(()) }
    async fn apply_config(&mut self, update: ConfigUpdate) -> anyhow::Result<ApplyConfigOutcome>;
    async fn handle(&self, ctx: &C) -> anyhow::Result<HandleOutcome>;
    fn health(&self) -> PluginHealth { PluginHealth::healthy() }
}
```

`instance_id` 不属于插件 trait。默认实例 ID 来自 `meta().name`，需要同类型多实例时由 `Bot::with_plugin_as(...)` 或 `RuntimePluginEngine::push_as(...)` 指定。

这里最关键的不是 trait 名字，而是“命令插件、正则插件、状态插件，最后都进入同一个实例运行时”。

运行时对外暴露只读 `RuntimePluginSnapshot` 清单，用于控制面、CLI 诊断和健康检查。snapshot 包含 instance id、metadata、manifest、当前 lifecycle/config state 和 `health()`，但不会触发 `init()` / `start()` / `stop()` 等生命周期 hook。

## 2. 用 Handler 管线替代 commands/wildcard 二分法

旧分发模型是：

- 先匹配 command
- 再跑 wildcard

这个模型足够做 Demo，但不够支撑复杂插件生态。v0.2 把插件入口预编译成 `HandlerDecl`：

- `HandlerEventKind`
- command / prefix / regex rule
- `Permission`
- `Priority`
- `Block`
- `ConcurrencyPolicy`

建议的执行管线：

1. adapter 产出统一上下文
2. runtime 做能力补全与基础解析
3. router 根据事件类型粗筛 handler
4. rule 阶段做命令、前缀、正则、来源等过滤
5. permission 阶段做用户 / 群 / 平台能力校验
6. handler 执行业务
7. block 阶段决定是否终止后续 handler

直接收益：

- 正则不再只是预留字段
- 优先级、拦截、权限成为内建语义
- 消息命令、系统事件、后续调度事件能进入同一模型

## 3. 存储、会话、Cron 和观测性先不进核心

这些能力都更像“插件服务”而不是当前最小内核的一部分。它们需要先回答几个问题：

- 插件之间如何声明、发现和调用服务
- 服务生命周期归谁管理
- 状态、定时任务和指标如何按插件实例隔离
- 跨 adapter/source 的状态 key 如何定义

在这些问题没稳定前，提前把 `Store`、`Conversation`、`Scheduler` 或 `MetricsSink` 做成公开主路径，会让插件作者依赖一套很快会改的 API。

当前内核只保留必要的运行时上下文：

- 插件实例身份
- Bot / platform 身份
- adapter 提供的能力
- outbound sender

等插件互调模型明确后，再把存储、会话、Cron 和观测性作为可组合服务接回来。

## 4. 配置系统升级为运行时生命周期

当前保留 `desired/applied version` 语义，同时补充配置生命周期：

- `draft`
- `validated`
- `applied`
- `rejected`

运行时支持：

- `ConfigUpdate::dry_run(...)`
- `ConfigUpdate::new(...)`
- apply 成功后标记 applied version
- apply 失败时保留错误上下文

后续可以继续补：

- 类型化配置结构体
- schema
- 默认值
- 配置迁移器
- 配置校验器

## 5. 平台扩展从裸 `extension<T>()` 升级为能力协商

`extension<T>()` 可以保留，因为它简单直接。但通用插件不应该把平台下转型当主路径。

v0.2 增加能力声明：

- `Capability::ProactiveSend`
- `Capability::MessageDelete`
- `Capability::Reaction`
- `Capability::GroupModeration`
- `Capability::RichSegments`

插件可以声明：

- 必需能力
- 可选能力

运行时启动时协商，得到：

- 可启动
- 降级启动
- 启动失败

这样插件作者可以先在 manifest 层声明约束，再在 handler 中按能力接口使用。

启动前要求可以通过 `RuntimePluginEngine::validate_startup_requirements()` 读取结构化报告。报告列出每个失败插件缺少的 capability 和 service；`init_all()` 使用同一条预检路径，所以失败时不会部分初始化插件集合。

## 6. 单 Bot，多渠道来源回复

Ayiou 的 Bot 是开发者定义的业务实例，不是外部平台账号。一个 Bot 可以接多个 adapter/channel source。

核心原则：

- 不引入多 Bot registry
- 不引入 Supervisor / RuntimeGraph
- adapter 负责把事件归一化到同一个 runtime
- outbound sender 根据消息来源回复到原渠道
- 插件看到的是统一 `Context` / `MsgContext`

这让多渠道部署只是 I/O 接入问题，而不是内核对象图问题。

# Ayiou 自己该有的风格

## 静态清晰 + 运行时可治理

建议保留：

- 显式 trait
- 显式 handler 声明
- 显式 config lifecycle
- 显式 capability manifest

同时补足：

- 自动注入常用上下文
- 宏生成 handler 声明
- 内建 permission / config 管线

## 不是只有消息命令，而是统一事件内核

v0.2 应继续把事件抽象推进到统一内核，至少包含：

- `Message`
- `Notice`
- `Request`
- `Lifecycle`
- `Custom`

命令只是 `Message` 的一种 rule，不应继续主导整个插件契约。

## 不是只做 filter，而是做声明式执行约束

装饰器 filter 很好用，但 Ayiou 可以再往前一步，把过滤、权限、并发和节流统一进 handler 声明：

```rust
HandlerDecl::message_commands(["ban"], ["/"])
    .require_permission(Permission::group("admin"))
    .priority(20)
    .block(true)
```

# 建议中的 v0.2 拆分

## Phase 1: 统一插件模型

- 删除 `Plugin<C>` 与 `ManagedPlugin` 双轨
- 宏生成 `RuntimePlugin`
- `Bot` 直接注册 `RuntimePlugin`

## Phase 2: 建立 Handler 管线

- 引入 handler 声明
- 引入 rule / permission / priority / block
- router 从命令索引升级为 handler 索引

## Phase 3: 建立权限高层 API 与插件服务边界

- 加入最小可用 permission gate
- 明确插件服务如何声明、发现和调用
- 再接回存储、会话、Cron 和观测性

## Phase 4: 类型化配置与能力协商

- 插件 schema
- dry-run apply
- capability manifest
- degraded startup

## Phase 5: 单 Bot 多渠道闭环

- adapter source identity
- outbound 按来源回复
- 多 adapter 接入同一个 runtime
- 统一管理 API 只管理当前 Bot 的插件与配置

# 当前仓库里的具体调整建议

## 应保留

- 统一消息模型
- `Context` 作为通用主路径
- `desired/applied config version` 语义
- worker queue 的有界并发方向
- 单 Bot runtime engine

## 应继续推进

- 完整类型化配置 schema / migration / validator
- 更完整的 source-aware outbound 路由
- lifecycle/custom event 进入统一 handler 管线
- permission gate 从 demo 语义升级为真实 user/group/role 能力判断
- 插件互调模型稳定后，再引入存储、会话、Cron 和观测性

## 可以后做

- marketplace
- 完整 Web UI
- 沙箱执行
- 完整动态安装

# 一句话方案

v0.2 最核心的不是“再造一个宏系统”或“做多 Bot 控制平面”，而是把 Ayiou 收敛成：

- 一个 Bot runtime
- 一套插件实例模型
- 一条统一事件执行管线
- 一组配置生命周期和后续可扩展服务边界
- 一套按消息来源回复的多渠道 I/O 模型

这样既能吸收 NoneBot2 的内核优点，也能吸收 AstrBot 的产品体验优势，同时保留 Ayiou 自己应有的工程清晰度。
